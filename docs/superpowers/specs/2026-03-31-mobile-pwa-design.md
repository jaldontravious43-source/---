# iPhone 移动端与 PWA 体验扩展设计（保持网页端可用）

## 背景与目标

当前项目以桌面网页入口为主（`index.html` + `src/main-standalone.js`），并强依赖登录后开局。用户新需求是：

1. 维持现有网页端可正常使用。
2. 增加可在苹果手机上顺畅游玩的体验。
3. 目标形态接近 App：可添加到主屏幕并以独立窗口打开。
4. 游客可直接开局；登录用于排行榜能力。

## 范围

### In Scope

- 新增移动端入口页面与移动专用脚本。
- 实现移动端“点击屏幕发钩”交互。
- 改造认证门禁为“游客可玩、登录增强”。
- iPhone Safari 兼容适配（视口、安全区、触控与音频解锁）。
- 第二阶段新增 PWA（manifest + service worker）。
- 更新 `README.md` 与 `CHANGELOG.md`。

### Out of Scope

- 改造为原生 iOS 应用（Swift/Flutter 等）。
- 引入后端 API 网关或 SSR。
- 重写完整游戏核心引擎。

## 方案总览

采用两阶段方案：

1. **阶段一：移动端入口拆分（先保证好玩）**
   - 新建 `mobile.html` + `src/main-mobile.js`。
   - 复用当前核心玩法与资源体系。
   - 游客可直接开局；登录后提供排行榜能力。

2. **阶段二：PWA 包装（再提升 App 感）**
   - 新增 `manifest.webmanifest`、service worker。
   - 支持添加到主屏幕与基础离线缓存。

该顺序优先解决“手机上顺手玩”的核心体验，再增强安装形态。

## 架构设计

### 入口层

- 桌面入口保留：`index.html` -> `src/main-standalone.js`
- 新增移动入口：`mobile.html` -> `src/main-mobile.js`

### 能力分层

- 核心玩法层：钩子、目标、碰撞、计分、章鱼喷墨腐蚀逻辑。
- 平台交互层：桌面键盘输入 vs 移动触控输入。
- 账号能力层：游客基础游玩、登录后排行榜增强。

### 共享策略

- 优先复用已有函数与配置定义。
- 对桌面与移动不同的 UI/输入，仅在入口脚本分叉，尽量不复制玩法逻辑。

## 交互与数据流

### 移动发钩交互

- 用户点击画布任意位置：若钩子空闲则发射。
- 摆动方向逻辑保持现有实现，避免重学操作。

### 账号与权限

- 游客态：可开始、可完成一局、可看本局分数。
- 登录态：在游客基础上可提交成绩、查战绩、改昵称、登出。
- 认证服务不可用时：不阻塞游客开局，仅禁用登录增强功能。

### 排行榜提交

- 仅在“已登录 + 客户端可用”条件下提交。
- 游客局不提交，结算页明确提示“登录后参与排行”。

## iPhone 兼容策略

### 视口与安全区

- 使用移动端视口策略，避免 iOS 地址栏伸缩导致布局跳动。
- 对 HUD 与按钮区域应用 `env(safe-area-inset-*)`。

### 触控与滚动

- 游戏画布区域禁用误触滚动与双击缩放干扰。
- 触控事件优先绑定 `pointerdown`（必要时补充 `touchstart` 兜底）。

### 音频解锁

- 首次用户触摸后统一解锁音频上下文。
- 音频失败静默降级，不影响游戏主流程。

## PWA 设计

### manifest

- 提供应用名称、图标、主题色、`display: standalone`。
- 推荐启动入口指向移动端页面（`mobile.html`）。

### service worker

- 缓存核心静态资源：HTML/CSS/JS/图片/音频。
- 使用版本化缓存键，发布新版本时自动清理旧缓存。
- 离线仅保障游客基础可打开与可玩，网络功能按可用性降级。

## 错误处理

- SDK/CDN 加载失败：显示“认证服务不可用”，保留游客开局。
- 短信发送失败：只影响登录流程，不阻塞游客游戏。
- 成绩提交失败：提示失败，不中断当前回合/结算。
- service worker 注册失败：不阻断页面，记录控制台告警。

## 测试与验收

### 手工验收

- iPhone Safari 访问移动入口可游客开局。
- 点击屏幕可发钩，整局流程可完成。
- 登录后可提交成绩；登出后仍可游客开局。
- 添加到主屏幕后可独立窗口打开。

### 自动化回归

- 现有 `npm test` 全通过。
- 为新增可测分支补最小逻辑测试（游客门禁、提交条件判断）。

## 风险与权衡

- 双入口会增加维护面，但能显著降低移动端改动对桌面的风险。
- PWA 在 iOS 上能力受 Safari 限制（例如后台行为），但“像 App 使用”可满足。
- 若未来功能继续扩张，应逐步抽离共享核心模块，减轻入口脚本耦合。
